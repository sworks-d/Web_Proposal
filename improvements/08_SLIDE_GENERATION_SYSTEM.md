# 08: 提案書生成システム設計

## 概要

分析システム（Web_Proposal）の出力データをもとに、提案書草案を自動生成する**別システム**。
Gensparkなどの汎用AIスライドツールを超える精度を目指す。

---

## システム構成

```
┌─────────────────────────────────────────────────────────────┐
│              分析システム（現Web_Proposal）               │
│  ─────────────────────────────────────────────────────────  │
│  目的：CDが提案を作るための「判断材料」を整理する          │
│  出力：構造化JSON + UI表示 + PDF                          │
│  これ単体で完結する                                       │
└─────────────────────────────────────────────────────────────┘
                         │
                         │ 出力JSONを渡す（任意）
                         ▼
┌─────────────────────────────────────────────────────────────┐
│            提案書生成システム（本設計）                    │
│  ─────────────────────────────────────────────────────────  │
│  目的：分析データをもとに「提案書草案」を自動生成          │
│  出力：pptx草案 / 構造化スライドデータ                    │
│  分析システムのオプション拡張                             │
└─────────────────────────────────────────────────────────────┘
```

## Gensparkとの差別化

| 観点 | Genspark | 本システム |
|---|---|---|
| **リサーチの深さ** | Web検索で自動収集 | **AG-01〜03で徹底分析済み** |
| **インサイト** | なし（表面情報のみ） | **AG-04 Sonnetで本質を掘る** |
| **コンセプト・コピー** | 自動生成（汎用的） | **SG-02 Opusで刺さる言葉** |
| **ストーリー設計** | 構成は自動だが緩急なし | **SG-03 Opusで緩急・伏線** |
| **ターゲット逆算** | なし | **ペルソナ・ジャーニーから設計** |
| **トーン選択** | テンプレート選択のみ | **Simple/Rich/Popで明確に差別化** |
| **重点配分** | できない | **どこを厚くするか選べる** |
| **pptx品質** | レイアウト崩れがち | **編集可能な状態で出力** |

### 精度の違い

**Gensparkの構成（汎用的）**
```
Ch.1 会社概要
Ch.2 課題
Ch.3 解決策
Ch.4 スケジュール
→ 論理的だが「刺さらない」
```

**本システムの構成（インサイト起点）**
```
Ch.1 あなたが今感じている違和感（共感）
Ch.2 その違和感の正体（インサイト）
Ch.3 だからこう変わるべき（方向性）
Ch.4 具体的にはこうする（設計）
Ch.5 実現すると何が変わるか（ビジョン）
→ 感情を動かして「刺さる」
```

### コピーの違い

| レベル | 例 |
|---|---|
| Genspark | 「採用力を強化する」「理想の人材を獲得」 |
| 本システム（Opus） | 「"安定"という言葉が、言い訳になっていませんか」 |

## 入力パラメータ

### 分析データ（必須）
- 分析システムの出力JSON全体

### ユーザー選択（UI）

#### 1. 提案書種別
| 種別 | 説明 | 章構成 |
|---|---|---|
| フル提案 | 新規制作・フルリニューアル | 課題→分析→ターゲット→ジャーニー→コンセプト→設計→IA→コンテンツ→KPI |
| 戦略提案 | コンペ初期・方向性合意 | 課題→ターゲット→インサイト→コンセプト→実現イメージ |
| 分析提案 | 現状分析・改善方向提示 | 現状分析→競合分析→ユーザー行動→課題構造→方向性 |
| コンテンツ提案 | コンテンツ追加・拡充 | コンテンツ課題→ターゲット×コンテンツ→戦略→サイトマップ→ページ設計 |
| 改善施策提案 | 部分改善・CV改善 | 問題点→課題優先順位→施策一覧→施策詳細→期待効果 |

#### 2. 枚数（10-50枚）
スライダーで選択

#### 3. 重点章（最大2つ）
- □ 課題・現状認識
- □ 分析・競合
- □ ターゲット・ジャーニー
- □ インサイト・コンセプト
- □ 設計・IA

選択した章は**+60%**のページ割り当て

#### 4. デザイントーン
| トーン | 参考 | 特徴 |
|---|---|---|
| **Simple** | Apple | 白ベース、余白大、1スライド1メッセージ、サンセリフ |
| **Rich** | FAS | ダーク基調、情報量多め、セリフ系混在、高級感 |
| **Pop** | 東組採用 | 明るい多色、イラスト多用、丸ゴシック、遊び心 |

#### 5. プレゼン相手
| 相手 | 特徴 |
|---|---|
| 経営層 | 結論先行、数字重視、簡潔 |
| 担当者 | 詳細説明、根拠重視、網羅的 |
| クリエイター | ビジュアル重視、感性訴求 |

## エージェント構成

### SG-01: 構成設計
**モデル**: Sonnet
**役割**: スライド構成・章立て

**入力**:
- 提案書種別
- 枚数
- 重点章

**出力**:
- 章リスト（順序・枚数配分）
- 各スライドの役割定義

---

### SG-02: コンセプト生成（最重要）
**モデル**: **Opus**
**役割**: コピー・キーメッセージ

**入力**:
- AG-04のインサイト
- ターゲットの言語（どんな言葉を使うか）
- 競合との差別化ポイント

**出力**:
- キーメッセージ（1行）
- サブコピー（2-3行）
- 各章の見出しコピー

**プロンプト方針**:
```
あなたは提案書のコピーライター。
以下のインサイトを「相手の心に刺さる言葉」に変換してください。

【禁止】
- 「〜を実現」「〜を強化」などの抽象的な動詞
- 業界用語・専門用語
- 競合も言いそうなこと

【必須】
- ターゲット本人が使う言葉で書く
- 「そう、それ」と思わせる具体性
- 違和感や葛藤を言語化する
```

---

### SG-03: ストーリー設計（最重要）
**モデル**: **Opus**
**役割**: 流れ・緩急・伏線

**入力**:
- SG-01の章リスト
- 重点章の指定
- プレゼン相手

**出力**:
- 章の順序（入れ替え含む）
- 各章の役割（導入/展開/山場/締め）
- 緩急設計（情報量の配分）
- フック設計（各章の冒頭で何を言うか）

**プロンプト方針**:
```
あなたはプレゼンの構成作家。
以下の素材を「相手が最後まで聞きたくなる流れ」に再構成してください。

【構成原則】
- 冒頭は「相手の現状認識」から入る（共感）
- 中盤で「意外な視点」を提示（驚き）
- 山場は「だからこうすべき」（納得）
- 締めは「実現した未来」（期待）

【避けること】
- 冒頭からの自社紹介
- データの羅列
- 結論の先延ばし
```

---

### SG-04: 本文生成
**モデル**: Sonnet
**役割**: 各スライドの文章

**入力**:
- SG-02のコピー
- SG-03の構成
- 分析データ（該当部分）

**出力**:
- 各スライドの見出し
- 本文（箇条書き or 文章）
- 補足説明

---

### SG-05: ビジュアル指示
**モデル**: Haiku
**役割**: 図解・写真の方向性

**入力**:
- SG-04の本文
- デザイントーン

**出力**:
- 各スライドのビジュアル指示
  - 図解の種類（フロー/マトリクス/グラフ等）
  - 写真の方向性（あれば）
  - アイコンの使用

---

### SG-06: スタイル適用
**モデル**: Haiku
**役割**: トーン反映・最終調整

**入力**:
- SG-04, SG-05の出力
- デザイントーン
- プレゼン相手

**出力**:
- 最終スライドデータ（JSON）
- pptx生成用の構造化データ

## デザイントーン詳細

### Simple（Apple参考）
```css
/* カラー */
--bg: #FFFFFF;
--text: #1D1D1F;
--accent: #0071E3;

/* タイポグラフィ */
font-family: "SF Pro", "Hiragino Sans", sans-serif;
font-weight: 300-400;

/* レイアウト */
- 余白: 大（左右15%以上）
- 1スライド1メッセージ
- 写真は大きく、テキストは少なく
```

### Rich（FAS参考）
```css
/* カラー */
--bg: #1A1A1A;
--text: #F5F5F5;
--accent: #C9A86C;

/* タイポグラフィ */
font-family: "Noto Serif JP", "Georgia", serif;
font-weight: 400-500;

/* レイアウト */
- 余白: 中（左右10%程度）
- 情報量多め、複数セクション可
- 質感のある写真、詳細なデータ
```

### Pop（東組参考）
```css
/* カラー */
--bg: #F8F8F8;
--text: #333333;
--accent: #FF6B35;
--secondary: #4ECDC4;

/* タイポグラフィ */
font-family: "Rounded Mplus 1c", "Hiragino Maru Gothic", sans-serif;
font-weight: 500-700;

/* レイアウト */
- 余白: 少なめ（賑やか）
- イラスト・アイコン多用
- 動きのあるレイアウト
```

## 出力形式

### JSON構造
```json
{
  "metadata": {
    "title": "〇〇株式会社 採用サイトリニューアル提案書",
    "type": "full",
    "slideCount": 25,
    "tone": "rich",
    "audience": "executive"
  },
  "concept": {
    "keyMessage": "\"安定\"という言葉が、言い訳になっていませんか。",
    "subCopy": "迷っている時間も、あなたのキャリアです。"
  },
  "slides": [
    {
      "id": 1,
      "chapter": "opening",
      "role": "hook",
      "title": "今、御社の採用サイトに起きていること",
      "body": ["..."],
      "visual": {
        "type": "photo",
        "direction": "現状サイトのスクリーンショット"
      },
      "notes": "プレゼンター用メモ"
    }
  ]
}
```

### pptx出力
- pptxgenjs（Node.js）を使用
- テンプレート（Simple/Rich/Pop）を事前作成
- JSON→pptx変換処理

## 実装優先度

| 優先度 | 内容 | 理由 |
|---|---|---|
| 1 | SG-02（コピー生成）プロンプト精査 | ここが一番差がつく |
| 2 | SG-03（ストーリー設計）緩急ロジック | 「聞きたくなる」設計 |
| 3 | pptx出力の品質担保 | 編集可能な状態で |
| 4 | トーン反映の精度 | Simple/Rich/Popの差 |
| 5 | UI実装 | 選択肢の操作性 |
