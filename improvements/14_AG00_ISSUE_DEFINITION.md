# AG-00: 課題定義エージェント

## 概要

AGパイプラインの最初に実行。クライアントの「問題」を整理し、解くべき「課題」を定義する。

---

## 目的

- 問題（現象・症状）から課題（解くべきテーマ）を導く
- CDが課題を選択・編集できる
- 追加情報をQA形式で収集し、後続AGの精度を上げる

---

## フロー

```
問題入力（自由記述・複数可）
    │
    ▼
AG-00 課題定義（Sonnet）
    │
    ├─ 課題候補（3-5個）
    ├─ 課題別QA
    └─ 共通QA
    │
    ▼
CD選択・入力画面
    │
    ├─ 課題を選択（複数可）
    ├─ 課題を編集（任意）
    └─ QAに回答（回答 / わからない / 不要）
    │
    ▼
AG-01〜07（選択課題 + 追加情報を元に分析）
```

## 処理ステップ

### STEP 1: 問題の分類・整理

入力された問題を以下の軸で分類：

| カテゴリ | 例 |
|---|---|
| brand | 競合との差別化、印象、認知 |
| content | 情報の過不足、伝え方 |
| ux | 使いやすさ、CV導線、離脱 |
| target | 誰に向けているか、ペルソナ |
| tech | 速度、更新しやすさ、CMS |

### STEP 2: 問題間の因果関係分析

どの問題が原因で、どの問題が結果か。

```
例：
「ターゲットが曖昧」→「誰にも響かない」→「応募が少ない」
     ↑原因                               ↑結果
```

因果ツリーを構築し、根本原因を特定。

### STEP 3: 課題の仮説生成

根本原因から「解くべき問い」を生成。

```
問題：応募が少ない
原因：ターゲットが曖昧
課題：「誰に、何を、どう伝えるか」を定義する必要がある
      → 課題名：ターゲット×メッセージの不在
```

### STEP 4: インパクト・難易度の評価

各課題を評価：

**インパクト（解決した場合の効果）**
- 高：KPIに直結（CV、売上、応募数）
- 中：体験改善（満足度、回遊）
- 低：部分最適（特定ページのみ）

**難易度（解決に必要なリソース）**
- 高：戦略変更、大規模リニューアル
- 中：コンテンツ追加、設計変更
- 低：文言修正、UI改善

### STEP 5: QA生成

各課題について、精度を上げるために必要な追加情報を質問形式で生成。

### STEP 6: 出力整形

3-5個の課題候補を生成。

## 出力

```typescript
interface Ag00Output {
  // 問題の整理
  problems: {
    id: string
    text: string                    // 元の問題文
    category: 'brand' | 'content' | 'ux' | 'target' | 'tech'
    isRoot: boolean                 // 根本原因かどうか
    causedBy?: string[]             // この問題を引き起こしている問題ID
    causes?: string[]               // この問題が引き起こしている問題ID
  }[]

  // 因果関係
  causalTree: {
    rootCauses: string[]            // 根本原因の問題ID
    symptoms: string[]              // 症状（結果）の問題ID
    chains: { from: string; to: string }[]
  }

  // 課題候補（3-5個）
  issues: {
    id: string
    title: string                   // 課題名（15字以内）
    description: string             // 説明（50字以内）
    rootProblems: string[]          // 関連する問題ID
    impact: 'high' | 'medium' | 'low'
    impactReason: string            // インパクトの根拠
    difficulty: 'high' | 'medium' | 'low'
    difficultyReason: string        // 難易度の根拠
    solvingApproach: string         // 解決アプローチの方向性
    agFocus: string[]               // この課題で重視すべきAG
  }[]

  // 課題別QA
  additionalQuestions: {
    issueId: string
    questions: {
      id: string
      question: string              // 質問文
      why: string                   // なぜこの情報が必要か
      example?: string              // 回答例
    }[]
  }[]

  // 全課題共通QA
  commonQuestions: {
    id: string
    question: string
    why: string
    example?: string
  }[]

  // AIの推奨
  recommendation: {
    priorityIssues: string[]        // 優先すべき課題ID
    reason: string                  // 推奨理由
  }
}
```

## QA回答の状態

| 状態 | 意味 | AIの動き |
|---|---|---|
| 回答 | 情報を入力 | その情報を使う |
| わからない | 必要だが情報がない | AIが推測・仮説を立てる |
| 不要 | この観点は分析しない | 触れない |

```typescript
interface QuestionAnswer {
  questionId: string
  status: 'answered' | 'unknown' | 'not_needed'
  answer?: string  // statusが'answered'の場合のみ
}
```

## 後続AGへの影響

選択された課題に基づいて、各AGの実行を調整：

```
CDが課題A（ターゲット不在）、課題B（差別化欠如）を選択した場合：

AG-02-STP      → 重点実行（課題Aに必要）
AG-02-POSITION → 重点実行（課題Bに必要）
AG-02-VPC      → 重点実行（課題Aに必要）
AG-02-JOURNEY  → 通常実行

AG-03-COMPETITOR → 重点実行（課題Bに必要）
AG-03-NIELSEN    → 軽量実行（選択されていない）

AG-04-INSIGHT  → 選択された課題にフォーカスしてインサイト生成
```

## プロンプトへの反映

後続AGのプロンプトに以下を追加：

```markdown
## 選択された課題

以下の課題を解決することに焦点を当ててください。

1. ターゲット×メッセージの不在
   → 「誰に」「何を」伝えるかが曖昧なため、誰にも響かない

2. 比較検討での差別化欠如
   → 競合と同じことを言っており、選ばれる理由がない

## 追加情報

【回答あり】
- Q: 来てほしい人は？
  A: 建築学科の学生。地元志向。

【わからない → 推測が必要】
- Q: 競合に勝てるポイントは？
  → ヒアリング情報と業界データから推測してください。
  → 出力には「※推測」とマークしてください。

【不要 → 分析対象外】
- Q: 過去の採用成功例
  → この観点は今回の分析では扱わない
```

## 課題候補の例

### 課題A: ターゲット×メッセージの不在

```
説明: 「誰に」「何を」伝えるかが曖昧なため、誰にも響かない

関連する問題:
└─ 応募が少ない、知名度が低い

インパクト: ◉◉◉ 高
└─ ターゲットが明確になれば、CV率が直接改善する

難易度: ◉◉○ 中
└─ ペルソナ設計とメッセージ再定義が必要

解決アプローチ:
└─ AG-02-STPでターゲット定義 → AG-04でメッセージ設計

重視するAG: AG-02-STP, AG-02-VPC, AG-04-INSIGHT
```

### 課題B: 比較検討での差別化欠如

```
説明: 競合と同じことを言っており、選ばれる理由がない

関連する問題:
└─ 古臭い、競合に負けている感がある

インパクト: ◉◉◉ 高
└─ 比較検討フェーズで選ばれる確率が上がる

難易度: ◉◉◉ 高
└─ ポジショニング戦略の再定義が必要

解決アプローチ:
└─ AG-03-COMPETITORで競合分析 → AG-02-POSITIONで差別化軸設計

重視するAG: AG-03-COMPETITOR, AG-02-POSITION, AG-04-INSIGHT
```

### 課題C: CV導線の断絶

```
説明: サイトに来ても「次に何をすべきか」がわからず離脱している

関連する問題:
└─ 情報が多すぎる、離脱が多い、応募が少ない

インパクト: ◉◉○ 中
└─ 来訪者のCV率は改善するが、流入数には影響しない

難易度: ◉○○ 低
└─ 導線設計とCTA改善で対応可能

解決アプローチ:
└─ AG-02-JOURNEYでユーザー行動分析 → AG-07C-4でCV導線設計

重視するAG: AG-02-JOURNEY, AG-03-NIELSEN, AG-07C-4
```

## 課題タイプ別のQA例

| 課題タイプ | 質問 | なぜ聞くか |
|---|---|---|
| ターゲット不在 | 「来てほしい人」はどんな人？ | ペルソナ設計の起点 |
| | その人は今どこで情報収集してる？ | タッチポイント設計 |
| | 採用で「この人は良かった」という例は？ | 成功パターンの言語化 |
| 差別化欠如 | 競合はどこ？（社名/URL） | 競合分析の精度 |
| | 競合に負けた理由は？ | 負けパターンの把握 |
| | 「これだけは負けない」と思う点は？ | 強みの言語化 |
| CV導線断絶 | 現在のCV数は？ | 定量ベースライン |
| | 離脱が多いページは把握してる？ | 問題箇所の特定 |
| | CVまでに何ステップある？ | 導線の複雑さ把握 |
| 情報設計 | 一番見てほしいコンテンツは？ | 優先順位の把握 |
| | 「これは要らない」と思うコンテンツは？ | 削除候補の特定 |
| ブランド印象 | 「こう見られたい」イメージは？ | トーン&マナーの方向性 |
| | 絶対避けたいイメージは？ | NGラインの把握 |

## モデル

**Sonnet**

理由:
- 構造化タスク（問題→課題）
- CDが編集可能なので、AIが完璧である必要がない
- パイプライン最初なので、ここで重くすると全体が遅い

## 実装優先度

Phase 1（MVP）:
- 問題入力UI
- AG-00実行（課題候補生成）
- 課題選択UI
- QA回答UI（3状態）
- 後続AGへの引き継ぎ

Phase 2:
- 課題の編集・追加
- 因果関係の可視化
- 推奨理由の詳細表示
