# 03: ターゲット逆算設計の強化

## 現状の問題

### 根本的な問題
- ペルソナ・ジャーニー・比較軸が**全て推定ベース**
- 実際のユーザー行動データとの接続がない
- 「こういうターゲットだろう」という仮説が検証されないまま設計に使われる

### 結果として起きていること
- 提案の根拠が「業界の一般論」や「経験則」に依存
- クライアントに「なぜそう言えるのか」と問われると答えられない
- 他社の提案と差別化できない

---

## 改善方針

### Tier 1: 検索ベースの検証（すぐ実装可能）

web_searchを使って、ターゲットの行動・心理を実データで裏付ける。

### Tier 2: 外部データ連携（将来実装）

Google Analytics、Search Console等との連携。
現時点ではスコープ外だが、設計は意識しておく。

---

## 改善1: AG-02-STP（セグメンテーション・ターゲティング）

### 現状の問題

```markdown
# 現状のプロンプトの問題点
- ペルソナを「作成」しているが「検証」していない
- 「転職意欲の高い層」等の分類が実データなし
- 案件種別（採用/BtoB/EC等）で決め打ち
```

### 追加すべきタスク

```markdown
## Task: ターゲット検証（web_search使用）

### Step 1: 検索行動の確認
以下のクエリで実際の検索行動を確認する：

採用案件の場合：
- "{業界名} 転職 理由"
- "{職種名} キャリア 悩み"  
- "{会社名} 転職 口コミ"
- "{業界名} 転職 年齢"

BtoB案件の場合：
- "{サービスカテゴリ} 比較"
- "{サービスカテゴリ} 選び方"
- "{課題名} 解決方法"

EC案件の場合：
- "{商品カテゴリ} 選び方"
- "{商品カテゴリ} 口コミ"
- "{商品カテゴリ} 比較 おすすめ"

### Step 2: 検索結果からの抽出
検索結果から以下を抽出：
- 実際に検索されている言葉（ターゲットの言語）
- よく言及される悩み・ニーズ
- 比較検討時に使われている軸
- 決め手として挙げられている要素

### Step 3: ペルソナへの反映
Step 2の結果を踏まえて：
- ペルソナの「悩み」を実際の検索語で更新
- decisionCriteriaを実際の比較軸で更新
- 「想定」から「確認済み」に信頼度を更新
```

### 出力形式の追加

```json
{
  "targetValidation": {
    "searchQueries": ["実行した検索クエリ"],
    "confirmedPainPoints": [
      {
        "painPoint": "確認できた悩み・ニーズ",
        "evidence": "どの検索結果から",
        "frequency": "どれくらい言及されていたか"
      }
    ],
    "confirmedDecisionCriteria": [
      {
        "criterion": "確認できた比較軸",
        "evidence": "どの検索結果から",
        "priority": "high|medium|low"
      }
    ],
    "targetLanguage": [
      {
        "ourTerm": "我々が使う言葉",
        "theirTerm": "ターゲットが使う言葉",
        "source": "どの検索結果から"
      }
    ]
  }
}
```

## 改善2: AG-02-JOURNEY（カスタマージャーニー）

### 現状の問題

- 各フェーズの設計が「想定」ベース
- 「この訪問者はこう動くだろう」という仮説
- 離脱理由やバリアーが一般論

### 追加すべきタスク

```markdown
## Task: ジャーニー検証（web_search使用）

### Step 1: 離脱理由の検索
以下のクエリで実際の離脱理由を確認：

採用案件の場合：
- "{会社名} 転職 やめた理由"
- "{業界名} 採用サイト 見にくい"
- "採用サイト 離脱 理由"
- "{業界名} 応募 ためらい"

BtoB案件の場合：
- "{サービス名} 導入 見送り 理由"
- "{サービスカテゴリ} 稟議 通らない"
- "{サービス名} 比較検討 決め手"

### Step 2: 成功パターンの検索
以下のクエリで成功パターンを確認：

- "{競合名} サイト わかりやすい"
- "{業界名} 採用サイト 好評"
- "{サービス名} 導入 決め手"

### Step 3: バリアーの検証
想定しているバリアーが実際に言及されているか確認：
- 想定バリアーのキーワードで検索
- 言及頻度を確認
- 言及されていないバリアーは優先度を下げる
```

### 出力形式の追加

```json
{
  "journeyValidation": {
    "confirmedBarriers": [
      {
        "barrier": "確認できたバリアー",
        "phase": "どのフェーズで発生",
        "searchEvidence": "どの検索から",
        "frequency": "言及頻度"
      }
    ],
    "unconfirmedBarriers": [
      {
        "barrier": "確認できなかったバリアー（想定のまま）",
        "confidence": "low",
        "note": "検索では言及なし、仮説として扱う"
      }
    ],
    "discoveredBarriers": [
      {
        "barrier": "検索で新たに発見したバリアー",
        "searchEvidence": "どの検索から",
        "addedToPhase": "どのフェーズに追加"
      }
    ]
  }
}
```

## 改善3: AG-03-COMPETITOR（競合分析のdecisionCriteria）

### 現状の問題

- decisionCriteria（ターゲットの比較軸）が想定ベース
- 「この業界ならこういう比較軸だろう」という一般論

### 追加すべきタスク

```markdown
## Task: 比較軸の検証（web_search使用）

### Step 1: 比較記事の検索
以下のクエリで実際の比較軸を確認：

- "{業界名} 比較 選び方"
- "{サービスカテゴリ} 比較表"
- "{会社名} vs {競合名}"
- "{サービス名} メリット デメリット"

### Step 2: 比較軸の抽出
検索結果の比較記事から：
- 実際に使われている比較軸
- 各軸の重要度（どれだけ言及されているか）
- クライアントの強み・弱みが言及されているか

### Step 3: decisionCriteriaの更新
- 実際に使われている軸を追加
- 使われていない軸の優先度を下げる
- 各軸のweightを言及頻度で更新
```

## 改善4: 新規エージェント「AG-02-VALIDATE」

### 目的

STP・ジャーニー・競合分析の**検証専門**エージェント。
各エージェントの出力を検索で検証し、信頼度を更新する。

### 位置づけ

```
AG-02-STP → AG-02-JOURNEY → AG-02-VALIDATE（検証）→ AG-03-COMPETITOR
```

### プロンプト骨子

```markdown
# AG-02-VALIDATE ターゲット設計検証担当

## Role
AG-02-STPとAG-02-JOURNEYの出力を検証し、
推定ベースの情報を検索で確認または否定する。

## Input
- AG-02-STPの出力（ペルソナ・セグメント）
- AG-02-JOURNEYの出力（ジャーニー・バリアー）

## Tasks

### Task 1: クリティカル情報の特定
提案に大きく影響する情報を特定：
- ターゲットの主要な悩み・ニーズ
- decisionCriteria（比較軸）
- 主要バリアー

### Task 2: 検索検証（20回上限）
各情報について検索で確認：
- 言及頻度を確認
- 言及されている言葉を確認
- 優先度の妥当性を確認

### Task 3: 信頼度更新
検証結果に基づき：
- confirmed: 検索で確認できた → high
- partial: 一部確認できた → medium  
- unconfirmed: 確認できなかった → low（仮説扱い）
- contradicted: 矛盾発見 → 修正必要

## Output
{
  "validationResults": {
    "persona": { ... },
    "painPoints": { ... },
    "decisionCriteria": { ... },
    "barriers": { ... }
  },
  "recommendations": [
    "設計前に確認すべきこと"
  ],
  "updatedConfidence": "high|medium|low"
}
```

## 改善5: ターゲット言語マッピング

### 問題

提案書やサイトコンテンツが「企業の言葉」で書かれがち。
ターゲットが使う言葉と乖離している。

### 新しい出力項目

各エージェントに以下を追加：

```json
{
  "languageMapping": [
    {
      "concept": "概念（例：採用情報）",
      "companyTerm": "企業が使う言葉（例：採用情報、キャリア）",
      "targetTerm": "ターゲットが使う言葉（例：求人、募集）",
      "searchEvidence": "確認した検索クエリ",
      "recommendation": "サイトでは「○○」を使うべき"
    }
  ]
}
```

### 活用方法

- ナビゲーションラベルの設計
- ページタイトル・見出しの設計
- CTAの文言設計
- SEOキーワード設計
