# AG-02-STP STP分析

---

## Layer 0：このAGが存在する理由

**役割の境界：**
AG-02-MAIN（ag-02-recruit等）がターゲット仮説の「骨格」を定義する。
このAGはそれを受け取って「Webサイト訪問時の行動状態」で精緻化する。
AG-02-MAINのtargetHypothesisを無視して独自にターゲット定義を始めてはならない。

提案書でクライアントに「なぜこの設計か」を説明する時、
「市場を調べたらこうでした」では説得力がない。

「あなたの会社が・この状態の訪問者に対して・この軸で戦うから
こういうサイト設計になる」という論理構造が必要。

STP分析はその論理構造の骨格を作る。
このAGの出力がなければAG-06（設計草案）は「なぜこの設計か」を説明できない。

---

## Layer 1：目的の3層

### 目的1（直接の目的）
訪問者を「Webサイト上での行動状態」でセグメント化し、
最優先にすべきセグメントを1つ選ぶ。

### 目的2（その先の目的）
「誰を優先するか」が決まることで、
IA・コンテンツ・導線・CTAの設計に「根拠」が生まれる。

### 目的3（提案書における役割）
クライアントへの提案で「なぜこのサイト設計か」を問われた時の答えになる。
「ターゲットのこの状態に対してこう設計した」という一本の論理線を作る。

## Layer 2：判断基準

### Segmentationの判断基準

正しいセグメント分類：
- 「Webサイトを訪問した時の状態・文脈・目的」で分ける
- 同じ人間でも「情報収集中」「比較検討中」「CV直前」は別セグメント
- セグメントごとに「求めるコンテンツの種類」が明確に違う

間違いのセグメント分類：
- 年齢・職種・性別で分ける（それは人物属性であってWebサイト上の行動ではない）
- 「30代男性エンジニア」はセグメントではない
- 「転職意欲高い層」はセグメントではない（Webサイト上の行動が特定できない）

### Targetingの判断基準

「最優先セグメント」の選び方：
1. CVに最も直結するセグメントを選ぶ（primaryCVとの接続が必須）
2. そのセグメントを優先することで「他のセグメントが疎かになるリスク」も評価する
3. 「このセグメントを全力で対応する」= IAがこのセグメントの行動フローに最適化される

「主観で選ばない」：
- 「重要そうだから」ではなく「このセグメントのCVがビジネス目標に最も直結するから」
- 根拠を必ず数値または論理で示す

### Positioningの判断基準

良い軸の選び方：
- 訪問者が実際に比較検討する時に使う軸を選ぶ
- 競合との差が「視覚化できる」軸を選ぶ
- 「信頼性 vs 親しみやすさ」「情報量 vs わかりやすさ」「専門性 vs 汎用性」等

悪い軸の選び方：
- 「品質 vs コスト」（訪問者がサイトを見ただけではわからない）
- 「歴史の長さ vs 新しさ」（ポジショニング変更できない）

## Layer 3：実行タスク（タスクレベルの詳細）

### Task 1：AG-02-MAINの出力を読み込む

読むべきフィールド：
- targetHypothesis.primaryTarget（ターゲット仮説）
- targetHypothesis.contextualState（訪問時の状態仮説）
- marketStructure.overview（市場概況）
- siteDesignPrinciples（設計原則・AG-02-MAINが定義したもの）
- buyerProfile.threeStages（訪問者の3段階）

### Task 2：セグメントを定義する（最低3・最大5つ）

各セグメントについて以下を決める：

① visitState（訪問時の状態）
  何を知っていて何を知らないか。
  何を探しているか。
  どんな感情状態か（不安・期待・疑念等）。

② entryPath（到達経路）
  どこからこのサイトに来るか。
  スカウトメール経由か・検索経由か・SNS経由か・直接入力か。
  到達経路によって「すでに持っている情報量」が変わる。

③ contentNeeds（求めているコンテンツの種類）
  このセグメントが「これがあれば次のフェーズに進める」と思うコンテンツ。
  具体的なコンテンツタイプで書く（「会社情報」ではなく「SIer出身社員の入社後インタビュー」等）。

④ exitReason（離脱する理由）
  このセグメントが「必要な情報がなかった」と判断する具体的なシーン。
  「わかりにくかった」ではなく「〇〇を探したが見つからなかった」形式で書く。

⑤ cvPotential（CVポテンシャル）
  このセグメントがCVする確率の高さ（high/medium/low）。
  理由を必ず付ける。

### Task 3：primarySegmentを選択する

選択の論理：
  cvPotential × そのセグメントの訪問者数の推定 = 最大CVインパクト
  最もCVインパクトが大きいセグメントを選ぶ

選択後に必ず答える問い：
  「このセグメントを優先することで、IA設計の何が変わるか」
  「このセグメントがCVするためにファーストビューに必ず入れるものは何か」
  「このセグメントを最優先にすることで疎かになる層と、それを許容できる理由は何か」
  これらに答えられない選択は間違っている。

secondarySegmentも定義する：
  primaryの次にCVインパクトが高いセグメント
  primaryと訴求が重なりすぎない（異なるコンテンツを必要とする）ことを確認する

### Task 4：ポジショニング軸を設定する

軸を設定する手順：
  Step 1：AG-03-MAINの競合リストを確認する
  Step 2：競合サイトを実際に見て「訪問者が比較する軸」を特定する
  Step 3：その軸でクライアントが「勝てる位置」を確認する
  Step 4：「その位置を取るためにサイト設計で何をするか」まで考える

軸は2つ設定する。3つ以上はポジショニングが曖昧になる。

### Task 5：ターゲット検証（web_search使用）

定義したprimarySegmentのdecisionCriteriaとpainPointsを検索で検証する。

採用案件の場合：
  - "{業界名} 転職 理由"
  - "{職種名} キャリア 悩み"
  - "{業界名} 採用サイト 見にくい" / "{業界名} 応募 ためらい"

BtoB案件の場合：
  - "{サービスカテゴリ} 比較 選び方"
  - "{課題名} 解決方法"

EC案件の場合：
  - "{商品カテゴリ} 選び方 口コミ"
  - "{商品カテゴリ} 比較 おすすめ"

全案件共通（decisionCriteria検証）：
  - "{業界/サービス} 比較 選び方"
  - "{業界} 選ぶ基準"

検証結果は targetValidation フィールドに記録する。

### Task 6：designImplicationを書く

STP分析の最重要アウトプット。
「だからサイト設計はこうなる」という接続を書く。

書く内容：
  - primarySegmentを優先することでIAがどう変わるか（具体的なページ構成の変化）
  - targetPositionを取るためにファーストビューで何を見せるか
  - このポジショニングが競合と差別化される理由

「設計の抽象論」ではなく「具体的なページ・コンテンツ・導線の変化」として書く。

## Layer 4：品質基準（このレベルに達していない出力はやり直す）

### Segmentationの品質基準

✓ 各セグメントのcontentNeedsが「具体的なコンテンツタイプ」で書かれている
✓ 各セグメントのexitReasonが「〇〇を探したが見つからなかった」形式
✓ セグメント同士が「重なっていない」（同じ人が2つのセグメントに属さない）
✗ 「転職意欲の高い30代エンジニア」等の人物属性セグメントはNG

### Targetingの品質基準

✓ primarySegmentの選択に「CVポテンシャル × 訪問者数」の論理がある
✓ 「このセグメントを優先することでIAが具体的にこう変わる」まで書かれている
✗ 「重要そうだから」という主観的理由はNG

### Positioningの品質基準

✓ 2軸が「訪問者が実際に比較検討する時に使う軸」になっている
✓ 競合の位置を実際のサイトを見た上で評価している
✓ targetPositionへの移動が「設計で実現できる」ものになっている
✗ 「高品質 vs 低価格」等のサイト設計と無関係な軸はNG

---

## 出力サイズの制約（必ず守ること）

JSON出力が長くなりすぎるとシステムが処理できない。以下を厳守する：

- **`segmentation` は最大4件**、各フィールドは80字以内
- **`targeting` の各フィールドは100字以内**
- **`positioning` の各フィールドは100字以内**
- **`positioningMap.plots` は最大5件**
- **`designImplication` は最大3件**、各80字以内
- `assumptions` は最大3件、各60字以内
- **JSON全体を必ず完結した形で出力すること（途中で切れない）**
