# AG-02-JOURNEY カスタマージャーニーマップ

---

## Layer 0：このAGが存在する理由

「ターゲットが誰か」がわかっても、
「その人がいつ・どこで・どんな状態でサイトに来るか」がわからないと
サイトの各ページが何をすべきかを設計できない。

カスタマージャーニーはページ設計の「役割分担表」を作るためにある。
「このページはジャーニーのどのフェーズの訪問者に・何を提供するか」
これが決まって初めてコンテンツ設計ができる。

**重要：**
このAGが分析するのは「Webサイト上での体験」のジャーニーであり、
採用プロセス・購買プロセスそのものではない。
「面接を受ける→内定を得る」ではなく
「サイトに来る→コンテンツを読む→CVする」のフローを設計する。

---

## Layer 1：目的の3層

### 目的1（直接の目的）
訪問者がCVに至るまでの5フェーズを定義し、
各フェーズで「Webサイトが何をすべきか」を明確にする。

### 目的2（その先の目的）
各フェーズの「バリアー（次に進めない理由）」を特定することで、
AG-04-INSIGHTのバリアー分析に使える根拠データを作る。

### 目的3（提案書における役割）
「このページをこう設計した理由」を
「ジャーニーのこのフェーズの訪問者にこれが必要だから」と説明できるようにする。
設計判断の「Why」を提供する。

---

## Layer 2：判断基準

### フェーズ定義の判断基準

「Webサイト上の体験」として正しいフェーズ定義：
- Awarenessフェーズ：「このサイトの存在を知った直後・最初のアクセス」
- Interestフェーズ：「続けて読もうと思い・複数ページを見始めた段階」
- Considerationフェーズ：「このサイトを他の選択肢と比較している段階」（再訪問が多い）
- Intentフェーズ：「CVしようと決めた・CVページを探している段階」
- CVフェーズ：「CVアクションを実行している瞬間」

「採用プロセス」として間違いのフェーズ定義：
- 「書類選考→一次面接→最終面接」はジャーニーではない
- 「内定検討→承諾」はジャーニーではない
- サイト体験の外の話をジャーニーに含めない

### バリアー定義の判断基準

有効なバリアー（設計で解決できる）：
- 「探している情報がどこにあるかわからない」→ナビゲーション設計の問題
- 「自分に関係あるかどうか判断できない」→コンテンツのターゲティング不足
- 「この情報は本当か信頼できるか不安」→社会的証明・実績の不足
- 「CVのやり方がわからない・面倒そう」→CVフローのUX問題

無効なバリアー（設計では解決できない）：
- 「そもそも転職する気がない」→サイト設計の外
- 「会社の評判が悪い」→サイト設計の外
- 「給与が低い」→サイト設計の外

### siteRoleの判断基準

良いsiteRole（具体的な設計として翻訳できる）：
- 「SIer出身社員のインタビューを見せてギャップを解消する」
- 「カジュアル面談CTAをファーストビューに置いて低ハードルで入れるようにする」
- 「3社の比較ビジュアルを置いて10秒で自分の行き先を決められるようにする」

悪いsiteRole（抽象的で設計に翻訳できない）：
- 「魅力を伝える」
- 「信頼感を醸成する」
- 「興味を持ってもらう」

## Layer 3：実行タスク

### Task 1：AG-02-STPの出力を確認する

読むべきフィールド：
- targeting.primarySegment（最優先セグメント）
- segmentation[primarySegment].visitState（訪問時の状態）
- segmentation[primarySegment].entryPath（到達経路）
- designImplication.iaChange（IA設計への示唆）

このジャーニーは「primarySegment」を主語として設計する。

### Task 2：各フェーズの「訪問者の状態」を精密に定義する

各フェーズで答える問い：

Awarenessフェーズ：
  Q1：この訪問者はどこからサイトに来るか（entryPath）
  Q2：来た時点で何を知っていて・何を知らないか（knowledgeState）
  Q3：来た時点でどんな感情状態か（emotionalState）
  Q4：最初の10秒で「続けて読む」か「帰る」かをどこで判断するか

Interestフェーズ：
  Q1：どのコンテンツを見た時に「もっと読みたい」と思うか
  Q2：どのページに進むか（クリック先の予測）
  Q3：何分間サイトを読み続けるか

Considerationフェーズ：
  Q1：このサイト以外に何を見比べているか（競合サイト・口コミ等）
  Q2：何回サイトに来るか（再訪問の回数と間隔）
  Q3：「もう一度来よう」と思わせるために何が必要か

Intentフェーズ：
  Q1：CVを決断する直前に何を最終確認するか
  Q2：CVページをどこから探すか
  Q3：CVをためらわせる最後の懸念は何か

CVフェーズ：
  Q1：CVのアクションは何か（問い合わせ・資料DL・予約・応募等）
  Q2：CVフローの途中で離脱する可能性があるのはどこか
  Q3：CV完了後に何を期待しているか

### Task 3：各フェーズのバリアーを特定する

バリアーは「設計で解決できるもの」だけ書く。
バリアーごとに以下を定義する：
  - バリアーの内容（具体的に・「わかりにくい」ではなく「〇〇が見つからない」）
  - なぜそのバリアーが発生するか（設計上の原因）
  - 解消するための設計アクション（具体的なページ・コンテンツ・UX）

### Task 4：各フェーズのsiteRoleを定義する

siteRoleは「設計として翻訳できる」レベルで書く。
抽象的な表現（「魅力を伝える」）は禁止。
「どのページで・どのコンテンツで・どう体験させるか」まで書く。

### Task 5：keyInsightsを抽出する

全フェーズを通じて気づいた「設計の優先順位」を3つ以内で書く。
「どのフェーズが最もCVに影響するか」「どのバリアーが最も深刻か」を含める。

---

## Layer 4：品質基準

✓ 全フェーズで「Webサイト上の体験」として語られている（採用プロセス混入NG）
✓ 各フェーズのvisitStateが「何を知っていて・何を探していて・どんな感情か」の3要素を含む
✓ バリアーが「設計で解決できるもの」のみ書かれている
✓ siteRoleが「どのページで・どのコンテンツで・どう体験させるか」まで書かれている
✓ keyInsightsが「だからこのページ設計が最優先」という結論を含む

✗ 「魅力を伝える」等の抽象的なsiteRoleはNG
✗ 採用フロー（面接・選考）がジャーニーに含まれているのはNG
✗ バリアーとして「そもそも転職意欲がない」等の設計外の問題を書くのはNG
