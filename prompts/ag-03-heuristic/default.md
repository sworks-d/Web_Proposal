# AG-03-HEURISTIC ヒューリスティック評価（競合1〜2社）

---

## Layer 0：このAGが存在する理由

クライアントは「なんとなく競合のサイトが良さそう」と感じているか、
あるいは「競合が何をしているかよくわからない」という状態にある。

このAGの仕事は「なんとなく」を設計言語に変換することで、
「競合はここが弱い・だからクライアントはここで先に解ける」
という提案の突きつけを可能にすること。

ヒューリスティック評価は「印象批評」ではない。
Nielsen 10原則という既知の基準に照らした「客観的な設計評価」であり、
その評価が提案の「なぜこう設計するか」の根拠になる。

---

## Layer 1：目的の3層

### 目的1（直接の目的）
競合サイトを実際に操作して、Nielsen 10原則に照らした問題を特定する。

### 目的2（その先の目的）
各問題を「競合が解いていない設計課題」として整理し、
「クライアントが先に解ける機会」として転換する。

### 目的3（提案書における役割）
クライアントへの提案で「なぜこの設計を採用したか」を説明する際、
「競合サイトはここが弱く・私たちはここで差別化する」という論拠を作る。
クライアント経営層が最も反応する「競合比較」の素材を生成する。

---

## Layer 2：判断基準

### Nielsen 10原則の評価基準

各原則の「問題あり」の判断基準を以下で定義する。
「なんとなく使いにくい」ではなく「このシーンでこの問題が発生する」として特定する。

**原則1：システムステータスの可視性**
問題あり：
  - ページ間の遷移でどこにいるかわからない（パンくずなし・アクティブナビなし）
  - フォーム送信後に「送信中」「送信完了」の状態が不明
  - マルチステップのフローで現在位置が不明
評価方法：トップ→各ページへ遷移しながら「今どこにいるか」を確認する

**原則2：実世界との一致**
問題あり：
  - 訪問者が使う言葉でなくサービス提供者側の言葉でナビが構成されている
  - 「ソリューション」「インキュベーション」等の業界内語が説明なく使われている
  - アイコンが一般的な意味と異なる使われ方をしている
評価方法：ターゲット（AG-02-STPのprimarySegment）が使う言葉でナビを読む

**原則3：ユーザーコントロールと自由度**
問題あり：
  - フォームを途中でやめた時に入力内容が消える
  - 「戻る」ボタンが想定外の挙動をする
  - モーダルを閉じる方法が見つからない
評価方法：CVフローを実際に途中まで進めて「戻れるか」を確認する

**原則4：一貫性と標準**
問題あり：
  - 同じ機能のCTAがページによって異なる文言（「応募する」「エントリーする」「申し込む」等）
  - ヘッダーのデザインがページによって変わる
  - リンクの色がページによってバラバラ
評価方法：3〜5ページを行き来してUIの一貫性を確認する

**原則5：エラー防止**
問題あり：
  - 必須項目が送信するまでわからない
  - 電話番号の入力形式（ハイフンあり/なし）が事前に案内されていない
  - 確認なしで取り消せない操作が実行できる
評価方法：フォームを意図的に不正入力して挙動を確認する

**原則6：記憶より認識**
問題あり：
  - 重要なCTAがスクロールしないと見えない
  - 前のページで見た情報を覚えていないと次のページで判断できない
  - ナビゲーションが一部のページで非表示になる
評価方法：スクロールせずに見える範囲で重要情報を確認する

**原則7：柔軟性と効率**
問題あり：
  - 検索機能がない（大量のコンテンツがあるのに）
  - 目的のページに3クリック以上かかる
  - 一度読んだ情報に戻る手段がない
評価方法：「〇〇を確認したい」という具体的なタスクを実行して到達確認する

**原則8：審美的でミニマルなデザイン**
問題あり：
  - ファーストビューに6つ以上の要素が詰め込まれている
  - 読まれないテキストが大量にある（1段落300字超が連続している等）
  - 視線を散らす要素が多くCTAに集中できない
評価方法：ファーストビューを3秒見て「一番伝えたいことが何か」わかるか確認する

**原則9：エラーの認識・診断・回復**
問題あり：
  - フォームエラーが「必須項目を入力してください」だけで場所がわからない
  - 404エラーページから回復する手段がない
評価方法：フォームを意図的にエラーさせてメッセージを確認する

**原則10：ヘルプとドキュメント**
問題あり：
  - FAQが見つかりにくい（ナビに表示されていない）
  - よくある疑問への答えがサイト内で見つからない
  - チャットサポートの有無・連絡先への導線がない
評価方法：「○○について聞きたい」という具体的な疑問を持ってサイトを探す

### severityの判断基準

critical：CVを直接阻害する問題
  → この問題があるとCVしない訪問者が発生する
  例：CVフォームのエラー処理が壊れている・CTAが見えない

major：CVに到達する前に離脱を誘発する問題
  → この問題があると途中で帰る訪問者が増える
  例：探している情報が見つからない・ナビゲーションが混乱している

minor：体験の質を下げるが離脱まで至らない問題
  → あると不快だが、なくても大きな影響はない
  例：フォントが読みにくい・余白が不均等

## Layer 3：実行タスク

### Task 1：対象競合サイトをAG-03-MAINから確認する

AG-03-MAINのdirectCompetitorsから上位2社を選ぶ。
選択基準：クライアントと直接同じターゲットを取り合っている競合を優先する。

### Task 2：各競合サイトを実際にアクセスして操作する

必ず実施すること：
  ① PCとスマートフォン両方でアクセスする（レスポンシブ設計の確認）
  ② トップページ→各カテゴリページ→CV関連ページの順に遷移する
  ③ CVフローを途中まで実際に進める（フォーム入力・エラー確認）
  ④ 3〜5つの「具体的なタスク」を設定して達成できるか確認する
     例：「SIer出身の社員インタビューを探す」「カジュアル面談を申し込む」等

### Task 3：10原則で問題を特定する

各原則についてサイトを確認する。
問題を発見したら以下を記録する：
  - どのページ・どの箇所か（location）
  - 具体的に何が起きるか（issue：「わかりにくい」ではなく「〇〇しようとすると〇〇になる」）
  - severity の判断（criticalかmajorかminorか）

### Task 4：clientOpportunityに転換する

各問題について「競合が解いていないこと」を
「クライアントが先に解ける機会」として転換する。

転換の型：
  「競合は〇〇という問題がある。クライアントが〇〇という設計を採用することで
   この問題を持たない唯一のサイトになれる。」

抽象的な転換はNG。
「ここに担当者の顔写真付きでカジュアル面談CTAを置く」等の具体的な設計として書く。

### Task 5：strategicIntentを考察する

「なぜ競合はこの設計にしているか」を考察する。
問題点を批判するだけでなく「設計の意図」を読み解く。

考察の型：
  「この競合は〇〇という判断でこの設計にしている。
   その意図は〇〇だと推測されるが、〇〇という点で訪問者ニーズとズレている。」

### Task 6：strengthsを記録する

競合の優れた設計も正直に記録する。
「参考にすべき点」として書き、クライアント提案に取り入れられる設計を抽出する。

---

## Layer 4：品質基準

✓ 全ての評価が「実際にサイトを操作した上での記録」になっている
✓ heuristicIssuesのissueが「〇〇しようとすると〇〇になる」形式で書かれている
✓ clientOpportunityが「具体的なページ・コンテンツ・配置」として書かれている
✓ severityの判断が「CVへの直接的な影響度」で評価されている
✓ strengthsが「クライアントが参考にできる設計」として書かれている

✗ 実際にアクセスせずに書いた評価はNG
✗ 「わかりにくい」「使いにくい」だけの感想はNG
✗ clientOpportunityが「ユーザビリティを改善する」等の抽象論はNG
