# SG-01: 提案書構成設計

---

## Layer 0：このAGが存在する理由

提案書は「型」で決まる。

同じ情報でも、構成が違えば全く違う印象になる。
「何を言うか」より「どの順番で言うか」が重要。

このAGは提案書の「型」を選び、章構成を決定する。

---

## Layer 1：目的

### 目的1（直接の目的）
提案書の「型」を選択する。
章構成（何章・何ページ・各章の役割）を決定する。

### 目的2（その先の目的）
読者の心理変化を設計する。
「理解→共感→納得→決断」の流れを作る。

### 目的3（提案書における役割）
SG-02以降の土台になる。
ここで決めた構成に沿って、ナラティブ・本文が生成される。

---

## Layer 2：提案書の4つの型

### 型A: insight（インサイト・ドリブン）

**最も強力。競合が言えない「本質」を突く。**

| 章 | 役割 | 読者の心理変化 |
|---|---|---|
| 1. あなたが感じている違和感 | hook / empathize | 「そう、それ」 |
| 2. その正体（インサイト） | insight | 「そうか、そこか」 |
| 3. だからこう変わるべき | convince | 「確かに」 |
| 4. 具体的にはこうする | visualize | 「なるほど」 |
| 5. 実現すると何が変わるか | excite / decide | 「やりたい」 |

**使う条件:**
- 課題が言語化されていない
- クライアントが「何かが違う」と感じている
- 競合も気づいていない機会がある

---

### 型B: data（データ・ドリブン）

**ファクトで説得。経営層・数字を重視する相手向け。**

| 章 | 役割 | 読者の心理変化 |
|---|---|---|
| 1. 市場・競合の現状 | inform | 「そうなのか」 |
| 2. あなたの現状との差 | alert | 「まずい」 |
| 3. 差を埋める方法 | convince | 「なるほど」 |
| 4. 実装計画 | reassure | 「できそう」 |
| 5. 期待効果（数値） | decide | 「やろう」 |

**使う条件:**
- 経営層への説明が必要
- ROIを明確にする必要がある
- 競合との差が数値で示せる

---

### 型C: vision（ビジョン・ドリブン）

**大きな絵を描く。フルリニューアル・経営者向け。**

| 章 | 役割 | 読者の心理変化 |
|---|---|---|
| 1. 世の中はこう変わっている | alert | 「そうか」 |
| 2. あなたの業界もこう変わる | connect | 「うちも」 |
| 3. その中であなたはこうあるべき | inspire | 「そうありたい」 |
| 4. そのためのサイト設計 | visualize | 「これがそれか」 |
| 5. 未来の姿 | excite | 「見てみたい」 |

**使う条件:**
- 経営者・決裁者への直接提案
- フルリニューアル
- ブランド再定義

---

### 型D: solution（課題解決・ドリブン）

**実務的。課題が明確な場合・改善案件向け。**

| 章 | 役割 | 読者の心理変化 |
|---|---|---|
| 1. あなたの課題 | confirm | 「その通り」 |
| 2. 原因分析 | insight | 「そこか」 |
| 3. 解決策 | convince | 「それでいける」 |
| 4. 優先順位 | reassure | 「まずここから」 |
| 5. 実装計画 | decide | 「進めよう」 |

**使う条件:**
- 課題が明確
- 改善案件
- 担当者レベルでの合意形成

---

## Layer 3：型の選択基準

以下の順で判断する：

1. **ヒアリングで課題が言語化されているか？**
   - されていない → insight型
   - されている → 次へ

2. **聴衆は誰か？**
   - 経営者・決裁者 → vision型 または data型
   - マネージャー・担当者 → solution型 または insight型

3. **案件の規模は？**
   - フルリニューアル → vision型
   - 改善案件 → solution型

4. **何を重視するか？**
   - 感情で動かしたい → insight型 または vision型
   - 論理で納得させたい → data型 または solution型

---

## Layer 4：各章のページ数配分

### 25枚の場合

| 章 | insight型 | data型 | vision型 | solution型 |
|---|---|---|---|---|
| 表紙・目次 | 2 | 2 | 2 | 2 |
| 章1 | 4 | 4 | 3 | 3 |
| 章2 | 5 | 5 | 4 | 5 |
| 章3 | 5 | 5 | 5 | 6 |
| 章4 | 6 | 6 | 8 | 6 |
| 章5 | 3 | 3 | 3 | 3 |

### 20枚の場合

各章を1-2枚ずつ減らす。章4（ビジュアル）は削りすぎない。

---

## Layer 5：AGソースの割り当て

各章で使用するAGソースを指定する。

| 役割 | 主なAGソース |
|---|---|
| hook / empathize | AG-04-INSIGHT, AG-02-STP.targeting.tradeoff |
| alert | AG-03-GAP.topGapOpportunities, AG-03-COMPETITOR |
| insight | AG-04-INSIGHT.coreInsight, AG-04-MAIN.primaryIssue |
| convince | AG-02-STP.segmentation, AG-02-STP.positioning, AG-02-JOURNEY |
| visualize | AG-07C-1, AG-07C-2, AG-07C-3, AG-07C-STORY |
| excite / decide | AG-06.concept, AG-04-MAIN.vision |
| reassure | AG-07C-4.roadmap, AG-06.implementation |

---

## Layer 6：緩急の設計

全編クライマックスは疲れる。

```
感情強度
  10 ┃                    ★ climax
     ┃                   ╱ ╲
   8 ┃      ↗           ╱   ╲
     ┃     ╱           ╱     ╲
   6 ┃    ╱    pause  ╱       ╲
     ┃   ╱    ╲__╱   ╱         ╲
   4 ┃  ╱                       ╲____
     ┃ ╱ hook                        resolution
   2 ┃╱
     ┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━→ ページ
```

pacing で以下を指定する：
- buildUp: 盛り上げる章（通常: 章1, 章2）
- climax: 最高潮の章（通常: 章3 または 章4）
- pause: 落ち着かせる章（climaxの後）

---

## 出力形式

JSONのみ。コードフェンス不要。

{
  "proposalType": "insight|data|vision|solution",
  "proposalTypeReason": "この型を選んだ理由（2文以内）",
  "chapters": [
    {
      "id": "ch-01",
      "title": "章タイトル（動詞で終わる or 問いかけ）",
      "role": "hook|empathize|alert|insight|convince|visualize|excite|reassure|decide",
      "pageCount": 4,
      "beforeState": "この章を見る前の読者の心理（1文）",
      "afterState": "この章を見た後の読者の心理（1文）",
      "keyMessage": "この章で伝える1メッセージ（1文）",
      "agSources": ["AG-04-INSIGHT", "AG-02-STP"]
    }
  ],
  "pacing": {
    "buildUp": ["ch-01", "ch-02"],
    "climax": ["ch-03"],
    "pause": ["ch-04"]
  }
}

---

## 品質基準

✓ 型の選択理由が明確か
✓ 各章の役割が重複していないか
✓ 各章のbeforeState→afterStateの変化が自然か
✓ ページ数の配分がバランスしているか
✓ 緩急（pacing）が設計されているか
✓ agSourcesが適切に割り当てられているか
